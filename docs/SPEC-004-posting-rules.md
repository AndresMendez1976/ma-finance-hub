# SPEC-004: Posting Rules

**Status:** Specification; NOT implemented
**Date:** 2026-03-29
**Applies to:** MA Finance Hub — Finance SaaS / Standalone Platform

---

## Purpose

Posting rules define how inbound financial events are automatically translated into double-entry journal entries. Events may originate from user actions within the platform, from external systems integrated via API, or from batch imports.

## Concepts

### What is a posting rule?

A posting rule is a tenant-scoped configuration that maps:

```
(event_type, conditions) → (journal_entry template with debit/credit lines)
```

When the platform receives a financial event, it looks up the applicable posting rule(s) for that tenant and event type, then generates the journal entry accordingly.

### Double-entry principle

Every journal entry produced by a posting rule must satisfy:

```
SUM(debit_amounts) = SUM(credit_amounts)
```

This invariant is enforced at the database level (check constraint or application validation).

### Example (conceptual)

**Event:** `invoice.created` for Tenant 1, amount $500.

**Posting rule for Tenant 1, event `invoice.created`:**

| Line | Account | Debit | Credit |
|---|---|---|---|
| 1 | Accounts Receivable (1200) | $500 | |
| 2 | Service Revenue (4100) | | $500 |

## Posting rule structure (conceptual)

> This is a conceptual schema. No table exists yet.

```
posting_rules
├── id              bigint PK
├── tenant_id       bigint FK → tenants
├── event_type      varchar (e.g., 'invoice.created')
├── description     text
├── is_active       boolean
├── created_at      timestamptz
└── updated_at      timestamptz

posting_rule_lines
├── id              bigint PK
├── posting_rule_id bigint FK → posting_rules
├── account_code    varchar (reference to chart of accounts)
├── entry_type      enum ('debit', 'credit')
├── amount_source   varchar (field path in event payload, or formula)
├── created_at      timestamptz
└── updated_at      timestamptz
```

## Processing flow (conceptual)

```
1. Platform receives financial event (e.g., invoice.created, amount=500, tenant_id=1)
   Source: user action, API call from external system, or batch import
2. Look up active posting_rules WHERE tenant_id=1 AND event_type='invoice.created'
3. For each matching rule:
   a. Resolve amount_source against event payload
   b. Generate journal_entry header
   c. Generate journal_lines from posting_rule_lines
   d. Validate double-entry balance
   e. Persist journal_entry + journal_lines
4. Return success/failure
```

## Event sources

| Source | Description |
|---|---|
| Platform UI | User creates invoice, records payment, etc. directly in MA Finance Hub |
| External API | An integrated system (e.g., MA ChiroBill, INIP, or any third party) sends a financial event via the platform's API |
| Batch import | CSV/file upload or scheduled sync |

> The platform does not assume any specific external system. It exposes a generic event ingestion API that any authorized caller can use.

## OPEN DECISIONS

| # | Decision | Context |
|---|---|---|
| 1 | Rule storage format | DB rows (as above) vs JSON/YAML config vs DSL |
| 2 | Amount resolution | Simple field mapping vs expression engine vs scripting |
| 3 | Multi-line rules | Fixed templates vs dynamic line generation |
| 4 | Rule versioning | How to update rules without affecting historical entries |
| 5 | Default rules vs custom | Whether system provides default rules that tenants can override |
| 6 | Rule validation | How to validate a rule before activation (dry-run?) |
| 7 | Tax handling | Whether tax lines are generated by posting rules or separately |
| 8 | Multi-currency | Whether posting rules need to handle currency conversion |
| 9 | Effective dates | Whether rules have start/end dates for time-bounded applicability |

## What is implemented (as of Day 4)

Nothing. No posting rules table, engine, or logic exists.

## What is NOT implemented

- posting_rules table
